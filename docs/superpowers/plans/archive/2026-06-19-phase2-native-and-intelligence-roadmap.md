## Context

The app is a **mobile-first PWA** installed to an iPhone home screen (`display: standalone`, `viewportFit: cover`, safe-area insets, `apple-mobile-web-app-capable`). It's a personal health/productivity intelligence tool: Oura sleep/readiness sync, intake logging, AI-analyzed evening reflections, calendar ingest, weekly rollups, and a daily AI briefing generated by a Vercel cron job.

Phase 1 delivered the UX/CRUD layer (greeting, prompts, caffeine warning, delete, date picker, settings, streaks, health correlation card, weekly page). Phase 2 is about making it **feel like a real iPhone app**, deepening the **intelligence** (the app's actual purpose — "understand what affects performance, productivity, confidence, energy"), hardening the **backend**, and **polishing design**.

**Eventual goal (explicitly NOT in this cycle):** wrap the app as a native `.ipa` for iPhone 15 Pro Max to unlock HealthKit, widgets, Live Activities, and the Action Button. This roadmap is structured so everything built now is reusable inside that wrapper later. See **"Future: Native IPA Wrapper"** at the end for the forward-reference.

## Scope

**In scope (this cycle):** Phases 0–4 below — all achievable as a PWA + Next.js/Neon backend with no App Store involvement.

**Out of scope (documented for later):** native wrapper, HealthKit, home/lock-screen widgets, Live Activities, Dynamic Island, Action Button, StandBy. These need a Capacitor/native shell (final section).

## Guiding constraints (carry over from Phase 1)

- Single user: `USER_ID = 1` (`lib/jobs.ts`). Keep new tables `user_id`-scoped so multi-user is a later flip, not a rewrite.
- DB: `sql` tagged template from `lib/db.ts` — **parameterized only**, no string interpolation. Tables created via `extra-schema.sql` (run in Neon SQL editor) and/or runtime `ensureTable()` guards.
- Timezone: always `userTz()` from `lib/jobs.ts`; fallback `America/New_York`.
- UI: keep `mx-auto max-w-md` layout and the Tailwind design tokens (`text-ink/ink-2/ink-3`, `bg-surface/bg`, `border-line`, `bg-accent`, `rounded-card/rounded-control`, `shadow-card`).
- Files < 500 lines; split when approaching.
- No new npm packages unless strictly necessary (each phase notes required deps explicitly).
- Validate input at API boundaries; trust client state internally.

## Phase 1 — Native iPhone feel (PWA)

The single biggest "this is a real app" jump. All work here survives into the native wrapper.

### 1.1 Web Push notifications — **highest impact** ✅ (2026-06-19)
**Why:** iOS 16.4+ delivers Web Push to home-screen PWAs. Reuses the existing cron/briefing backend.

- **New table** `push_subscriptions(user_id, endpoint TEXT PK, p256dh TEXT, auth TEXT, created_at TIMESTAMPTZ)`.
- **New dep:** `web-push` (server-side VAPID signing). Generate VAPID keys once → env (0.5).
- **Client:** a Settings toggle that calls `Notification.requestPermission()`, subscribes via `registration.pushManager.subscribe({ applicationServerKey })`, POSTs the subscription to a new `app/api/push/subscribe/route.ts`. *iOS requires the prompt be triggered by a user gesture and the app be installed to the home screen.*
- **SW:** add `push` + `notificationclick` handlers to `public/sw.js` (open `/dashboard` or a deep link).
- **Server triggers** (extend `lib/jobs.ts` / new `lib/push.ts`):
  - Briefing ready (after `generateBriefing`) → "Your morning briefing is ready."
  - Readiness drop: in `runDailyJob`, compare today's `readiness_score` vs 7-day avg; if down ≥ 15, "Readiness is low today — protect your evening."
  - Reflection nudge: a second cron (e.g. 21:00 user-tz) that checks for a missing `reflections` row today and, if streak > 0, nudges to protect it. Add a `vercel.json` cron entry; reuse `CRON_SECRET` auth pattern from `app/api/cron/daily/route.ts`.
- **Effort:** L. **Risk:** Medium — iOS Web Push is finicky (must be installed PWA, HTTPS, correct VAPID). Test on the actual device early.

### 1.2 Home-screen badge
- Use the **Badging API** (`navigator.setAppBadge(n)` / `clearAppBadge()`), set from the SW on push and on app focus. Badge = count of actionable items (e.g. unlogged-reflection today). iOS support is partial but degrades gracefully.
- **Effort:** S. **Dependency:** 1.1 (SW + push).

### 1.3 Face ID / passkey login — replaces the `hello` password
**Why:** the current model (single shared password, and the session **cookie value literally equals `AUTH_SECRET`** — see `lib/auth.ts`) is weak. WebAuthn gives biometric unlock and removes the shared secret.

- **New table** `webauthn_credentials(user_id, credential_id TEXT PK, public_key BYTEA, counter BIGINT, created_at)`.
- **New dep:** `@simplewebauthn/server` (+ browser helper). Endpoints: `/api/auth/webauthn/register/{options,verify}` and `/api/auth/webauthn/login/{options,verify}`.
- **Migrate session** from "cookie value = secret" to a signed/opaque session token (random id stored server-side or a signed JWT with `AUTH_SECRET` as the signing key, not the value). Keep the password path as a fallback for first-time registration.
- **Effort:** L. **Risk:** Medium (touches auth — guard with the password fallback). **Note:** passkeys also carry into the native wrapper.

### 1.4 Web Share + Share Target
- **Share out:** Web Share API button on a reflection / briefing.
- **Share in:** add `share_target` to `manifest.json` so iOS share-sheet can send selected text into a "quick log / quick reflect" route. **Effort:** S–M.

**Phase 1 verify:** install to home screen on the iPhone 15 Pro Max; confirm a real push arrives from a manual trigger; Face ID round-trips; badge updates.

## Phase 2 — Health intelligence (the app's purpose)

### 2.1 Generalized correlation engine — **most valuable feature**
**Why:** Phase 1 hard-coded one alcohol↔sleep card (`lib/scores.ts buildCorrelationMessage`). Generalize it into a config-driven "what affects my X" system.

- **Approach (no new table needed):** a `lib/correlations.ts` with a declarative list of **factor → outcome** rules. Each factor is a SQL predicate over `intake_log` for day D; each outcome is an `oura_daily` metric for day D or D+1. Compute `AVG(outcome) WHERE factor present` vs `WHERE absent`, plus n-counts and a simple significance guard (min sample size, min delta).
- **Seed factors:** caffeine-after-14:00↔sleep_score; alcohol↔sleep_score & ↔hrv_avg; alcohol↔next-day readiness; late intake (any after 21:00)↔total_sleep_seconds. Outcome on D+1 via `oura_daily.day = DATE(intake) + 1`.
- **UI:** replace the single Health insight card with a ranked "Insights" list (only show factors with enough data + meaningful delta). Add a confidence/sample-size footnote ("based on 23 nights").
- **Effort:** M. **Risk:** Medium (SQL joins + avoiding spurious correlations — enforce min-n).

### 2.2 Semantic reflection search
**Why:** `VOYAGE_API_KEY` is already slotted; Neon supports `pgvector`.

- **Schema:** `CREATE EXTENSION vector;` then `reflection_embeddings(reflection_id PK, embedding vector(1024), model TEXT)` (Voyage `voyage-3` = 1024 dims; confirm).
- **Backfill + on-write:** embed `raw_text` on reflection save (extend `app/api/reflections/route.ts` POST) and a one-time backfill script in `scripts/`.
- **Query:** `/api/reflections/search?q=` → embed query, `ORDER BY embedding <=> $1 LIMIT k`. New search UI on the Reflect page.
- **Effort:** M. **Dependency:** Voyage key, pgvector enabled.

### 2.3 AI weekly/monthly narrative
- Extend the existing `weeklyRollup` (`lib/jobs.ts`) — you already call Claude for `notable_note`. Add a longer "month in review" generated monthly (on the 1st, like `isMonday` gate) that summarizes patterns + one concrete suggestion, stored in a `narratives` table or reusing `briefings`. Surface on the Weekly page.
- **Effort:** S–M (reuses `extractWithTool`, `lib/prompts.ts` tool pattern).

### 2.4 Goals / habits
- **Schema:** `goals(id, user_id, kind, target_json, active, created_at)` — kind ∈ {sleep_hours_gte, reflect_daily, caffeine_before_hour, …}. Adherence computed from existing tables; reuse `computeStreak` from `lib/dates.ts`.
- **UI:** a Goals section (Settings or a new tab — mind tab overflow, see 4.4) showing adherence % + streak per goal.
- **Effort:** M.

### 2.5 Briefing upgrade
- Fold calendar load + readiness into one actionable morning card on the dashboard ("Big day — 3 meetings, readiness 62. Protect your evening."). The data already flows through `generateBriefing` (`reflections`, `oura`, `upcoming_events`); enrich `BRIEFING_SYSTEM`/`briefingTool` in `lib/prompts.ts` to emit a single headline + 1–3 actions.
- **Effort:** S.

**Phase 2 verify:** seed/known data produces correct correlation deltas (unit-test the pure math like Phase 1 did for `buildCorrelationMessage`); semantic search returns relevant past entries; weekly narrative reads coherently.

## Phase 3 — Backend / data

### 3.2 Oura webhooks (push instead of cron-pull)
- Register Oura webhook subscriptions; new `/api/oura/webhook` verifies signature and triggers an incremental `syncOura`. Keep the daily cron as a backstop. **Effort:** M. **Dependency:** Oura app config (`OURA_CLIENT_ID/SECRET` already present).

### 3.3 Materialized aggregates
- Two materialized views (or summary tables) for dashboard/health/weekly so pages stop recomputing joins per request: `mv_daily_scores` (last 90d) and `mv_correlations` (factor deltas, refreshed by the daily job). Refresh inside `runDailyJob`. **Effort:** M.

### 3.4 Offline-first logging (background sync)
- Use the real SW (0.4): queue intake/reflection POSTs in IndexedDB when offline, replay via the Background Sync API (or on next launch — iOS lacks full Background Sync, so a focus-time flush is the reliable path). **Effort:** M. **Risk:** iOS SW limits — fall back to flush-on-foreground.

### 3.5 Intake expansion
- Broaden `intake_log.type` beyond caffeine/alcohol: workouts, mood, weight, meals. Optional manual entry; later auto-import from Apple Health export (CSV/XML) ahead of native HealthKit. **Effort:** M.

## Future: Native IPA wrapper (NOT this cycle — forward reference)

When ready to ship an `.ipa` for iPhone 15 Pro Max, the lowest-friction path is **Capacitor** wrapping this same Next.js app (or a static export of it), giving an Xcode project that builds a real binary.

**What the wrapper unlocks that a PWA cannot:**
- **HealthKit** — read Apple Health (steps, workouts, HR, sleep) directly instead of CSV import (supersedes 3.5's import path).
- **Home/Lock-screen widgets** & **StandBy** — a WidgetKit extension showing readiness/briefing (native Swift, fed by a shared API).
- **Live Activities / Dynamic Island** — e.g. a "wind-down" or focus timer.
- **Action Button** — map to "quick log" or "quick reflect" via an App Shortcut.
- **Richer notifications**, true background refresh, and biometric APIs (though WebAuthn passkeys from 1.3 already cover most auth).

**Build now to make that later cheap:**
- Keep all data access behind clean JSON API routes (already the pattern) so native widgets/extensions can call the same endpoints.
- Keep `user_id` scoping (multi-device/account later).
- Passkeys (1.3) and Web Push (1.1) both carry over.
- Avoid PWA-only assumptions in business logic; keep it in `lib/`.

**Open decisions for that cycle:** Capacitor vs. React Native rewrite vs. SwiftUI native (Capacitor recommended to reuse this codebase); Apple Developer account ($99/yr); push migration to APNs (or keep Web Push inside the WebView); HealthKit entitlements & privacy strings.
